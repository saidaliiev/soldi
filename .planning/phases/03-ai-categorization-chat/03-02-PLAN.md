    ```ts
    /**
     * Normalizes a raw merchant_name string to a deterministic key for exact-match lookups
     * in merchant_overrides (CONTEXT D-02).
     *
     * Algorithm:
     *   1. lowercase
     *   2. NFKD Unicode normalization (decomposes Latin diacritics into base + combining mark)
     *   3. strip combining marks (U+0300..U+036F)
     *   4. replace any character NOT in [a-z0-9, Cyrillic Ѐ-ӿ Ԁ-ԯ, whitespace] with a single space
     *   5. collapse consecutive whitespace to single space
     *   6. trim
     *   7. truncate to 64 chars (D-02 cap)
     *
     * Script coverage: Latin (after NFKD) + Cyrillic (Ukrainian/Russian) survive intact.
     * Other scripts (Arabic, Chinese, Hebrew, etc.) collapse to whitespace — acceptable
     * for v1 portfolio scope given Irish + Ukrainian ICP. Pseudonym-quality only — this
     * is not a security primitive.
     *
     * SOURCE OF TRUTH for normalization. Edge Function copy at
     * supabase/functions/_shared/normalize.ts MUST mirror this byte-for-byte (verified
     * by diff-gate in 03-02 verification + 03-01 verification).
     *
     * Idempotent: normalizeMerchantKey(normalizeMerchantKey(x)) === normalizeMerchantKey(x).
     */
    export function normalizeMerchantKey(raw: string): string {
      return raw
        .toLowerCase()
        .normalize('NFKD')
        .replace(/[̀-ͯ]/g, '')                  // strip combining diacritical marks
        .replace(/[^a-z0-9Ѐ-ӿԀ-ԯ\s]/g, ' ')  // keep Latin alnum + Cyrillic (Ѐ-ӿ + Ԁ-ԯ) + whitespace; everything else → space
        .replace(/\s+/g, ' ')
        .trim()
        .slice(0, 64);
    }
    ```

Together they preserve all Ukrainian + Russian + Belarusian merchant names verbatim. The combining-mark stripping uses explicit Unicode escapes (`̀-ͯ`) instead of the literal range syntax to ensure the source file is ASCII-safe and reproducible across editors / line-ending normalization.

    ```sql
    -- Phase 3 / 03-02: rewrite merchant_overrides from substring-match to exact-match on normalized merchant_key.

    -- 1. Stand up new schema in a temporary table
    CREATE TABLE merchant_overrides_v2 (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      merchant_key TEXT NOT NULL,
      category_id INTEGER NOT NULL REFERENCES categories(id),
      source TEXT NOT NULL CHECK (source IN ('user','llm','mcc')),
      confidence REAL NOT NULL,
      created_at INTEGER NOT NULL,
      updated_at INTEGER NOT NULL,
      UNIQUE(merchant_key)
    );

    -- 2. Migrate existing rows. The OLD schema (merchant_pattern + created_by_user) becomes:
    --    merchant_key = lowercase(trim(merchant_pattern)) -- best-effort; users who relied on substring matching may lose hits, but this is acceptable per CONTEXT D-02 (exact match is the new contract).
    --    source = 'user' if created_by_user = 1 else 'llm'
    --    confidence = 1.0 if source='user' else 0.85
    INSERT INTO merchant_overrides_v2 (merchant_key, category_id, source, confidence, created_at, updated_at)
    SELECT
      lower(trim(merchant_pattern)),
      category_id,
      CASE WHEN created_by_user = 1 THEN 'user' ELSE 'llm' END,
      CASE WHEN created_by_user = 1 THEN 1.0 ELSE 0.85 END,
      created_at,
      created_at
    FROM merchant_overrides
    WHERE length(trim(merchant_pattern)) > 0
    GROUP BY lower(trim(merchant_pattern));  -- dedupe on collisions; first-wins via implicit grouping

    -- 3. Swap tables
    DROP TABLE merchant_overrides;
    ALTER TABLE merchant_overrides_v2 RENAME TO merchant_overrides;

    -- 4. Index for exact-match lookups (UNIQUE constraint already implies an index but be explicit)
    CREATE INDEX idx_merchant_overrides_key ON merchant_overrides(merchant_key);
    ```

    Note: the lower(trim(...)) inside the migration is a SQL-side approximation of normalizeMerchantKey because op-sqlite does not run JavaScript inside a SQL statement. This is acceptable: collisions get deduped by GROUP BY; users will see at most a minor data-loss event on the first launch after Phase 3 (some overrides become rule-of-thumb keys that won't match new normalized lookups). Acceptable per the project-scope/portfolio philosophy in CLAUDE.md; document the trade-off in the migration's top comment.
