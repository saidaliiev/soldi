## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app → iOS notification center / lock screen | Notification bodies render on a locked device, visible without auth |
| jar ledger → notification scheduler | Contribution data drives notification content |
| app → iOS notification permission | Permission must be requested in-context, not silently at launch |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Information Disclosure | digest body on lock screen | mitigate | Digest amount is the single figure the user explicitly opted into (toggle, default OFF, in-context permission). Body carries one aggregate spend total, never per-merchant/per-transaction detail. Editorial phrasing, no account identifiers. ASVS L1: minimal sensitive data on notifications. |
| T-05-08 | Information Disclosure | milestone body on lock screen | mitigate | Body is `"<pct>% of the way there"` + jar name only — a percentage, never the absolute jar balance or target amount. |
| T-05-09 | Information Disclosure | notification error logging | mitigate | Catch logs `err.name` only; never jar name + amount or spend figures together (CLAUDE.md), `__DEV__`-gated. |
| T-05-10 | Tampering / DoS | scheduleDailyDigest re-entrancy (now called on every foreground) | mitigate | `scheduleDailyDigest` is idempotent: it ALWAYS `cancelScheduledNotificationAsync(DIGEST_NOTIFICATION_ID)` by a stable identifier before re-scheduling. Foregrounding the app repeatedly cancels-then-recreates the single 09:00 trigger — it can never stack duplicate notifications. The foreground re-schedule itself introduces no new high threat (no new input source, same opted-in aggregate body); the duplicate-notification DoS surface that the per-foreground call could create is fully mitigated by the stable-id cancel. |
| T-05-11 | Repudiation | permission requested silently at launch | mitigate | Permission requested only inside the DigestToggle enable path (in-context, D-03) — grep-gate asserts no `requestPermissionsAsync` at app launch. The foreground re-schedule calls `scheduleDailyDigest` only (no permission request), so it does not move the permission prompt off the toggle. |
| T-05-SC | Tampering | npm installs | mitigate | No new packages — expo-notifications already a dep (app.json plugin). No legitimacy checkpoint needed. |
</threat_model>
