<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dev machine -> Apple / EAS credentials | Apple ID / ASC API key / signing certs cross into build config and CI |
| eas.json -> source control | Apple identifiers must not be hardcoded secrets in the repo |
| app -> sentry/react-native (optional) | Sentry client may be absent (P0 #7) — init must not assume presence |
| TestFlight build -> internal testers | A distributed build exposes the full app surface to testers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-15 | Information Disclosure | Apple credentials in eas.json | mitigate | Apple ID / Team ID / ASC App ID are non-secret identifiers; signing certs/keys stay in EAS-managed credentials or env, never committed. Placeholder convention preserved until the human supplies values; no API keys written to the repo. |
| T-05-16 | Denial of Service | Sentry absent crash on init | mitigate | initObservability() no-ops (return, no throw) when EXPO_PUBLIC_SENTRY_DSN is absent — grep-gate asserts no unconditional throw on the DSN-absent path (P0 #7 graceful absence). |
| T-05-17 | Tampering | double Sentry init | mitigate | initObservability() is idempotent (guard flag) — repeated startup calls cannot double-instrument. |
| T-05-18 | Elevation of Privilege | wrong distribution (external/public) | mitigate | Profile is Internal distribution only; action explicitly forbids Apple beta App Review and external testers (D-04). Reduces exposure surface of the beta build. |
| T-05-SC | Tampering | npm installs | mitigate | No new runtime packages — sentry/react-native already a dep; eas-cli is a CLI tool invoked, not an installed package. No legitimacy checkpoint needed. |
</threat_model>
